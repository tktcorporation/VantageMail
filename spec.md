# VantageMail — プロダクト要件定義書 v1.0

**モダンなマルチGmail統合メールクライアント**

Version 1.0 | 2026-03-20 | ドラフト

技術スタック: Electrobun + Vite + React / Cloudflare Workers
Gmail API ネイティブ | デスクトップ & Web | AI依存なし

---

## 1. 課題定義

Gmailは最も普及したメールプロバイダだが、Googleはファーストパーティのデスクトップクライアントを提供しておらず、Web版は複数アカウント管理時のUXが悪い。既存のサードパーティ製クライアントは大きく2つに分かれる。Superhuman（$30/月）のようなプレミアム路線は個人には高すぎ、IMAP系クライアントはGmailネイティブの機能（ラベル、カテゴリ、Pub/Subによるリアルタイムプッシュ）を活かせない。

個人用・仕事用・副業用など2つ以上のGmailアカウントを使い分けるユーザーは、ブラウザタブを何枚も開くか、重量級のElectronアプリを使うか、GmailをただのIMAPとして扱う汎用クライアントで我慢するしかない現状がある。軽量でGmail API直結のデスクトップ＋Webクライアントは、サブスク課金なしでSparkレベルのUXを提供するものがまだ存在しない。

---

## 2. ゴール

1. **統合受信トレイ（Unified Inbox）** — 3つ以上のGmailアカウントを横断し、サブ秒でのアカウント切り替えとクロスアカウント検索を実現
2. **Gmail APIネイティブ統合** — ラベル、カテゴリ、プッシュ通知、検索演算子をフル活用
3. **デスクトップアプリ80 MB以下**（Electronの~270 MBと比較）、Electrobunによるコールドスタート500ms以下
4. **Web版** — Cloudflare Workers/Pagesにデプロイ可能、UIコードベースの95%以上を共有
5. **キーボードファーストUX** — Cmd+Kコマンドパレットで、Spark同等以上の操作速度を実現

---

## 3. スコープ外（Non-Goals）

- **AI機能（自動下書き、要約、分類）** — スコープを絞る。プラグイン層として後から追加可能な設計にする
- **IMAP / Outlook / Yahoo対応** — Gmail API専用。非Gmailはv2で検討
- **カレンダー統合** — Google Calendar APIは別スコープ。v2に先送り
- **チームコラボレーション（共有ドラフト、コメント、委任）** — 個人利用にフォーカス。v1ではチーム機能なし
- **モバイルアプリ（iOS / Android）** — デスクトップ＋Web優先。モバイルはPWAまたはネイティブで後日対応
- **セルフホスト型バックエンド** — 全状態はGmailに存在。Cloudflareは認証トークンとPushリレーのみ

---

## 4. ユーザーストーリー

### 4.1 マルチアカウントユーザー

- US-1. マルチアカウントユーザーとして、すべての受信トレイが1つのUnified Inboxに統合されていることで、アカウント間を行き来せずに重要なメールを見つけたい。
- US-2. マルチアカウントユーザーとして、どのアカウントからでも作成・返信でき、スレッドの受信アカウントに基づいて送信元が自動選択されることで、誤ったアドレスからの送信を防ぎたい。
- US-3. マルチアカウントユーザーとして、Gmailの検索演算子を使ったクロスアカウント検索で、どのアカウントに届いたメールでも見つけたい。

### 4.2 キーボードパワーユーザー

- US-4. パワーユーザーとして、Cmd+Kコマンドパレットからすべてのアクションを実行できることで、マウスに触れずにメール操作を完結させたい。
- US-5. パワーユーザーとして、プリセット切替（Spark / Gmail / Superhuman / カスタム）でショートカットを選べることで、以前のクライアントの筋肉記憶をそのまま使いたい。
- US-6. パワーユーザーとして、vim式のJ/Kナビゲーションとクイックアクション（E=アーカイブ, S=スヌーズ, #=ゴミ箱）で、受信トレイを最速で処理したい。

### 4.3 集中したいユーザー

- US-7. メールに圧倒されがちなユーザーとして、Smart Inboxグルーピング（人物 / 通知 / ニュースレター）で、まず本当の会話に集中したい。
- US-8. メールに圧倒されがちなユーザーとして、スヌーズと送信予約で、適切なタイミングにメールを先送り・予約送信したい。
- US-9. メールに圧倒されがちなユーザーとして、Gatekeeper機能で初回送信者を承認制にし、営業メールが受信トレイを汚さないようにしたい。

## 6. 技術アーキテクチャ

### 6.1 システム概要

VantageMailはシンクライアントアーキテクチャを採用し、Gmailが唯一の信頼できるデータソース（Source of Truth）となる。サーバー側にメールデータを一切保存しない。バックエンド（Cloudflare Workers）はOAuthトークン交換、Pub/Sub Webhookリレー、スヌーズ/送信予約のスケジューリングのみを担当する。

### 6.2 デスクトップアプリ: Electrobun

- **ランタイム:** Bun（メインプロセス）+ ネイティブWebView（レンダラー）
- **レンダラー:** システムWebView（macOS: WebKit, Windows: Edge WebView2, Linux: WebKitGTK）
- **UIフレームワーク:** React 19 + Vite 6（Webと共有）
- **IPC:** ElectrobunのビルトインRPC（BunメインプロセスとWebView間）
- **トークン保存:** OSキーチェーン（macOS Keychain, Windows Credential Manager, Linuxのlibsecret）
- **目標サイズ:** 80 MB以下（Electron相当の~270 MBと比較）
- **コールドスタート:** インタラクティブまで500ms以下

### 6.3 Webアプリ: Cloudflare Pages

- **ビルド:** 同一のReact + Viteコードベースから静的アセットをCloudflare Pagesに出力
- **コード共有:** 95%以上のUI共有。プラットフォーム固有コードは `/platform/desktop` と `/platform/web` アダプターに分離
- **トークン保存:** Web Crypto APIで暗号化したIndexedDB
- **プッシュ:** Web Push API + Service Workerによるバックグラウンド通知

### 6.4 バックエンド: Cloudflare Workers

- **OAuthプロキシ:** Google OAuth 2.0トークン交換とリフレッシュを処理。パブリッククライアント用PKCE。
- **Pub/Subリレー:** Gmailプッシュ Webhookを受信し、WebSocket（Durable Objects）またはWeb Push経由でクライアントにファンアウト。
- **スケジューラー:** Cloudflare Cron TriggersによるSnooze/Send Later。スケジュールメタデータをWorkers KVに保存。
- **メール非保存:** Workersはメール本文を一切参照しない。トークン参照、プッシュメタデータ、スケジュールタイムスタンプのみ。

### 6.5 Gmail API統合

- **プロトコル:** Gmail REST API v1（IMAPではない）。ラベル、カテゴリ、検索演算子、バッチ操作を有効化。
- **認証スコープ:** `gmail.modify`, `gmail.compose`, `gmail.labels`（gmail.readonlyではない — アーカイブ/ラベル操作にmodifyが必要）。
- **プッシュ:** `users.watch()` でCloud Pub/Subトピックを監視。Cron Triggerで7日ごとにre-watch。
- **Batch API:** Gmailバッチエンドポイントで一括アーカイブ/ラベル/ゴミ箱（1リクエストあたり最大100メッセージ）。
- **同期:** プッシュ通知後に `history.list()` でインクリメンタル同期。初回ロード時は `messages.list()` でフル同期。
- **レートリミット:** 250クォータユニット/ユーザー/秒。指数バックオフ付きクライアント側レートリミッター。

### 6.6 データフロー

```
[クライアント] <--WebSocket--> [CF Durable Object] <--HTTP--> [Google Pub/Sub] <--watch--> [Gmail API]
[クライアント] --REST--> [Gmail API]（直接。読み書き操作用）
[クライアント] --HTTPS--> [CF Worker: /oauth/*]（トークン交換のみ）
```

## 7. 主要アーキテクチャ判断

### 7.1 Gmail API vs IMAP

| Gmail API（採用） | IMAP（不採用） |
|---|---|
| ネイティブのラベル、カテゴリ、スター | 汎用的でGmail固有機能を喪失 |
| Gmailの検索演算子をフル活用 | フォルダベース（ラベルなし） |
| Pub/Subによるプッシュ通知 | ポーリングのみ（リアルプッシュなし） |
| バッチ操作（1リクエスト100件） | 個別操作（一括処理が遅い） |
| スレッドファーストなデータモデル | メッセージベース（スレッドはクライアント側構築） |

### 7.2 Electrobun vs Electron vs Tauri

| Electrobun（採用） | Electron | Tauri |
|---|---|---|
| バンドル ~64 MB | バンドル ~270 MB | バンドル ~10 MB |
| Bunランタイム（高速起動） | Node.js + V8 | Rustメインプロセス |
| ネイティブWebView | Chromiumをバンドル | ネイティブWebView |
| TypeScriptネイティブ | 成熟したエコシステム | 学習コストが高い |

Electrobun採用理由: TypeScriptネイティブなDX（Tauriの Rustと比較）、軽量バンドル（Electronと比較）、Bunの高速メインプロセス、2026年2月時点でv1のプロダクション準備完了。

## 9. 成功指標

### 9.1 先行指標（Week 1–4）

| 指標 | 目標値 | 計測方法 |
|---|---|---|
| コールドスタート時間 | インタラクティブまで500ms以下 | Lighthouse / 手動タイマー |
| スレッド表示レイテンシ | キャッシュ時200ms以下、ネットワーク時1s以下 | Performance.now() 計測 |
| 検索レスポンスタイム | クロスアカウント検索2秒以下 | APIタイミング + レンダリング指標 |
| プッシュ通知遅延 | Gmail受信から5秒以内 | Pub/Subタイムスタンプ差分 |
| アプリバンドルサイズ | デスクトップ80 MB以下 | ビルド出力の計測 |

### 9.2 遅行指標（Month 1–3）

- **日次アクティブ利用:** 初期セットアップ後、週5日以上アプリを起動
- **アカウント継続率:** 設定済みアカウントの90%が30日後もアクティブ
- **キーボード利用率:** 受信トレイ操作の50%以上がキーボードショートカット経由
- **Inbox Zero達成率:** Gmail Web版のベースラインと比較して3倍の頻度でInbox Zeroを達成

## 11. Google API規約・コンプライアンス要件

### 11.1 規約上の位置づけ

Gmail APIを使ったサードパーティ製メールクライアントの開発・提供は **Google APIの利用規約上、明確に許可されている**。Mimestream、Spark、Edison Mail等の既存クライアントも同様にGmail APIを使い、Googleの審査を通過して運用中。

ただし `gmail.modify` と `gmail.compose` は **Restricted Scope（制限付きスコープ）** に分類されており、一般公開には以下の審査プロセスが必須となる。

### 11.2 必要な審査プロセス

| ステップ | 内容 | 期間 | 費用 |
|---|---|---|---|
| **1. ブランド審査** | OAuth同意画面の検証（ロゴ、プライバシーポリシーURL、利用規約） | 2–3営業日 | 無料 |
| **2. CASA セキュリティ監査** | Google認定の第三者監査機関（App Defense Alliance）によるセキュリティ評価。データの安全な取扱い、削除要求への対応能力を検証 | 2–6ヶ月 | $500–$75,000（ティア・複雑さによる） |
| **3. 年次再審査** | Letter of Assessment（LOA）承認日から12ヶ月ごとにCASA再評価 | 上記と同様 | 上記と同様 |

### 11.3 準拠が必要なポリシー

- **[Google API Services User Data Policy](https://developers.google.com/terms/api-services-user-data-policy)** — ユーザーデータの取扱い、開示、保持、削除に関する包括的ポリシー
- **Limited Use（制限付き利用）要件** — 取得したデータをアプリの機能提供以外に使用しないこと。広告表示・データ販売・第三者提供の禁止
- **プライバシーポリシーの公開** — アプリがアクセスするデータの種類、利用目的、保持期間、削除方法を明記した公開ポリシーが必須

### 11.4 VantageMailにとっての有利な点

VantageMailのアーキテクチャはCASA審査を通過しやすい設計になっている:

- **サーバー側にメールデータを保存しない**（Cloudflare WorkersはOAuthトークンとスケジュールメタデータのみ）
- **メール本文をバックエンドが参照しない**（クライアント ↔ Gmail API の直接通信）
- **トークンはOS Keychain / Encrypted IndexedDBに暗号化保存**
- **データ削除要求への対応が容易**（サーバーにデータがないため、ローカルキャッシュの消去のみ）

### 11.5 開発フェーズ別の制約

| フェーズ | 審査状況 | 制約 |
|---|---|---|
| 開発・テスト | 不要 | テストユーザー100人まで。手動追加 |
| クローズドベータ | 不要 | テストユーザーを手動追加。「未確認アプリ」警告が表示される |
| 一般公開 | CASA必須 | 審査完了まで公開不可。未審査の場合「Google hasn't verified this app」画面が表示される |

### 11.6 マイルストーン

- **Phase 2開始時（Week 7）** — CASA監査機関の選定・見積もり取得
- **Phase 3（Week 11）** — プライバシーポリシー・利用規約の整備・公開
- **Phase 3完了時（Week 14）** — CASA申請提出（審査期間2–6ヶ月のバッファを確保）
- **Phase 4 ベータ公開前** — CASA承認の取得を確認
